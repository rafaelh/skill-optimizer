# PEP 723 inline metadata for agent tools

Apply PEP 723 when the script is **standalone** — meant to run in environments
that don't share the project's `pyproject.toml`. That includes:

- Anything in `bash/scripts/`
- Anything in `claude/skills/<name>/scripts/`
- One-off helpers a colleague might copy to another machine

**Skip** PEP 723 for modules that live inside a package (e.g. `src/tools/`,
`src/actions/`) where deps come from the project's `pyproject.toml` — the
inline block would just go stale alongside it.

## Block placement and shape

At module top, after the shebang and before the docstring:

```python
#!/usr/bin/env python3
# /// script
# requires-python = ">=3.11"
# dependencies = [
#     "httpx>=0.27",
#     "rich>=13",
# ]
# ///
"""Tool description..."""
```

## Rules

- **`requires-python` must match what the code actually uses.** Don't claim
  `>=3.8` if you use `match` (3.10), PEP 695 generics (3.12), or the new
  `type` statement (3.12). The block is a hard contract — `uv run` refuses
  to run on a Python that doesn't satisfy it, and an inaccurate floor makes
  the script silently break on older interpreters.
- **Use lower bounds (`>=X.Y`), not exact pins.** Exact pinning belongs in
  lockfiles, not standalone-script metadata. Pick the lowest version with
  the API surface you actually call.
- **Every non-stdlib import must appear in `dependencies`.** If you later
  add `import httpx`, append `"httpx>=…"` to the block — `uv run` installs
  only what's listed.
- **Don't mix PEP 723 with a package's `pyproject.toml`.** If the file is
  imported as a module from elsewhere in the same package, delete the inline
  block — `pyproject.toml` owns deps and the inline block will drift.

## Shebang choice

| Shebang                                  | When to use                                                                                                                |
|------------------------------------------|----------------------------------------------------------------------------------------------------------------------------|
| `#!/usr/bin/env -S uv run --script`      | You want the file directly executable (`./tool.py`) on machines without a pre-configured venv. uv resolves deps each run.  |
| `#!/usr/bin/env python3`                 | Tool is invoked as `python3 tool.py` from inside a project venv. The block documents intent even if it isn't doing the resolving. |

## Drift modes to watch for

- **`requires-python` drifts when you reach for newer syntax.** A block that
  claims `>=3.10` will pass validation today and then break on the next
  import that needs PEP 695 generics or `type` statements. Bump the floor
  in the same edit that introduces the syntax — don't leave it for "later".
- **`dependencies` drifts when you add an import.** A new `import httpx`
  without a matching `"httpx>=…"` line in the block means `uv run` succeeds
  on machines where httpx is already installed and fails on fresh ones —
  a portability bug masked by your dev environment. `validate_agent_tool.py`
  catches the obvious cases; treat its warnings as blockers, not noise.
